# 设置页与健康档案页 UI 一致性改造设计

## 背景

当前项目在 `public/src/pages/setting` 与 `public/src/pages/record` 下存在多套并行的页面视觉表达：

- `setting/Index.vue` 已部分接入项目现有的 `care` 风格体系。
- `setting/About.vue`、`setting/Account.vue`、`setting/Feedback.vue` 仍主要使用各自独立的渐变背景、局部按钮和局部导航样式。
- `record/Index.vue` 具备较强的数字人主视觉表达，但背景、底部卡片、弹窗和局部色阶与当前项目其他页面的一致性不足。

项目已经在 `public/src/styles/index.css` 中沉淀了可复用的页面风格基座：

- `care-page`
- `care-card`
- `care-cell-group`
- `care-primary-btn`
- `care-danger-btn`

本次改造以复用这套现有风格体系为核心，避免继续新增零散页面样式。

## 目标

- 让 `setting` 目录下页面与项目当前已有页面风格保持一致。
- 让 `record/Index.vue` 在保留数字人主视觉的前提下，收敛到项目统一的背景、卡片、按钮和弹窗语义。
- 减少页面级重复 CSS，优先复用已有全局样式能力。
- 不改动现有页面业务逻辑、交互路径和路由结构。

## 非目标

- 不调整路由、页面跳转和业务流程。
- 不重写 `record/Index.vue` 的主视觉布局，不移除数字人展示区。
- 不新增全局组件，不将本次工作扩展为通用设计系统重构。
- 不接入真实账号设置、反馈提交、设备管理等后端能力。

## 已确认方案

采用方案 A：

- `setting` 页面全面接入现有 `care` 风格。
- `record/Index.vue` 保留数字人主视觉，仅统一背景、底部卡片、按钮、弹窗、文字层级和局部配色。

## 设计原则

### 1. 统一背景语义

所有改造页面优先使用 `care-page` 作为页面背景基底，避免继续使用分散的单页渐变定义。

### 2. 统一卡片语义

卡片、表单容器、信息分组统一优先使用：

- `care-card`
- `care-cell-group`

使页面在圆角、毛玻璃透明度、边框和阴影层级上保持一致。

### 3. 统一操作语义

主操作使用 `care-primary-btn`，风险操作使用 `care-danger-btn`，避免单页自定义按钮视觉分裂。

### 4. 保留页面识别度

`record/Index.vue` 继续保留数字人展示区、名称编辑入口、属性指示器和四宫格功能入口，不改为与设置页同构的列表页。

### 5. 优先小步修改

仅对当前页面结构进行样式收敛，不做大规模拆组件，确保改动可控、易回归。

## 页面级设计

## `setting/Index.vue`

当前状态：

- 已使用 `care-page` 和 `care-cell-group`。
- 顶部介绍卡片仍是页面内联样式组合。

改造策略：

- 保持现有结构不变。
- 将顶部信息块收敛为 `care-card` 语义，使设置首页成为该目录的视觉基准页。
- 保留退出登录按钮的风险语义。

预期结果：

- 首页继续作为设置目录入口页。
- 视觉上与其余设置子页完全同族。

## `setting/Account.vue`

当前状态：

- 使用独立渐变背景。
- 列表容器和危险操作区域没有统一接入 `care` 体系。

改造策略：

- 页面根容器切换为 `care-page`。
- 头部增加与设置首页一致的说明性卡片，用于承接“账号安全/绑定信息”语义。
- 账号信息分组、风险操作分组统一使用 `care-cell-group`。
- “注销账号”保持红色风险语义，但样式收敛到项目现有危险操作体系。

预期结果：

- 页面与设置首页、反馈页、关于页在背景和卡片层级上统一。

## `setting/Feedback.vue`

当前状态：

- 使用独立背景渐变。
- 提交按钮使用单页 `neu-btn-primary` 自定义样式。

改造策略：

- 页面根容器切换为 `care-page`。
- 顶部增加说明性卡片，用于承接反馈页面标题、副标题和引导文案。
- 输入区域使用 `care-card` 或 `care-cell-group` 语义承载。
- 提交按钮改为 `care-primary-btn`，移除单页按钮样式。

预期结果：

- 提交页与全局主按钮表现统一。
- 页面结构和视觉层级更接近当前项目首页/设置页风格。

## `setting/About.vue`

当前状态：

- 使用独立渐变背景。
- Logo、企业信息、版权信息布局有效，但卡片容器和文本层级与其他设置页差异较大。

改造策略：

- 页面根容器切换为 `care-page`。
- Logo 区、简介区、企业信息区按 `care-card` 层级收敛。
- 保留现有文案内容、品牌展示与备案信息，不修改信息本身。
- 文本颜色统一为现有深浅层级，增强阅读一致性。

预期结果：

- 关于页仍保留品牌感，但整体与设置目录同一套视觉体系。

## `record/Index.vue`

当前状态：

- 页面已有明显主视觉，包含头像标识、姓名编辑、数字人背景、属性指示器和底部四宫格。
- 底部功能区使用单页玻璃态实现，和当前项目 `care-card` 语义并不一致。
- 弹窗和按钮样式与项目其余页面未完全对齐。

改造策略：

- 页面根容器改为 `care-page` 背景基调。
- 保留数字人展示区的排版、素材和信息结构。
- 底部功能区调整为更贴近 `care-card` 的圆角、边框、透明度和阴影层级。
- 四宫格功能入口的图标底板、文字颜色、容器间距统一到项目现有配色。
- 编辑姓名弹窗中的输入区和操作按钮收敛到项目按钮体系。
- 弱化单页“科技蓝”孤立感，使页面既有主视觉特色，又不脱离项目整体色板。

预期结果：

- 用户仍然能一眼识别“健康档案”是特色页面。
- 与设置页、我的页等页面切换时，整体风格不再割裂。

## 数据流与交互影响

本次改造不引入新的数据流。

现有交互保持不变：

- 返回上一页
- 退出登录
- 修改密码提示
- 注销账号确认
- 反馈内容提交
- 档案姓名编辑与本地存储写入
- 档案菜单点击提示

因此本次工作主要属于展示层和结构层的样式一致性改造。

## 错误处理

不新增新的异步流程或错误处理逻辑。

需要保持以下现有行为不变：

- 反馈内容为空时提示用户输入。
- 编辑姓名为空时提示用户输入。
- 注销账号前必须二次确认。
- 退出登录前必须二次确认。

## 实施范围

计划修改以下文件：

- `public/src/pages/setting/Index.vue`
- `public/src/pages/setting/Account.vue`
- `public/src/pages/setting/Feedback.vue`
- `public/src/pages/setting/About.vue`
- `public/src/pages/record/Index.vue`

如发现必要的轻量样式补充，也只允许在现有全局样式体系内做小范围增强，避免扩展到无关页面。

## 验证方案

本次按用户要求执行：

- `npm run lint`

同时进行静态核对：

- 页面是否仍能正常返回上一页。
- 页面按钮、列表、弹窗是否仍可正常触发原有逻辑。
- `record/Index.vue` 的数字人主视觉是否保留。
- 设置目录页面是否统一使用 `care` 风格语义。

## 风险与控制

风险点：

- 页面结构较多依赖 Tailwind 原子类，样式收敛时容易出现局部间距变化。
- `record/Index.vue` 视觉层较重，若收敛过度可能削弱原有辨识度。

控制方式：

- 保持 DOM 结构尽量稳定，优先替换容器语义与色板，不大改布局层级。
- `record/Index.vue` 只调整底色、卡片、弹窗和色阶，不移除主视觉模块。

## 结论

本设计聚焦于“统一已有视觉体系”，不是重新设计页面。

实施后预期达到的效果是：

- 设置页形成统一的 `care` 风格目录体验。
- 健康档案页保留特色主视觉，同时与项目整体 UI 语言一致。
- 页面级重复样式减少，后续维护成本下降。
